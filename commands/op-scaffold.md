# Scaffold an Operation From a Plan Doc

You are scaffolding a multi-plan operation from a plan doc. The user gives you a path to a plan markdown file (typically generated by an external Opus using `~/.claude/operation-templates/META_PROMPT_FOR_PLAN_OPUS.md`, or hand-written following the format spec below).

Your job: validate the plan doc, create a Plane ticket hierarchy, and output a ready-to-run `/tch` command. **You do not write any files to disk.** You do not run the operation.

## Input

The argument is a path to a plan markdown file. Examples:
- `/op-scaffold docs/proposals/auth-rewrite-plan.md`
- `/op-scaffold ~/Downloads/my-op-plan.md`

If no argument is given, ask the user for the plan file path.

## Step 1 — read and validate

1. Read the plan file end-to-end.
2. Run the 5 hard-reject checks below in order. If any fail, STOP, print all failures together, and exit. Do NOT offer a "proceed anyway" path.

### Validation — 5 hard rejects

**Hard reject 1 — top section:** First H1 is exactly `# Operation: <slug>` where slug is lowercase, kebab-case, ≤40 chars. File has a `## Why this exists` section.

**Hard reject 2 — multi-plan structure (skip if flat):** If a `## Dispatch order` table is present: table has Plan, Name, and Depends-on columns; every row in the table has a corresponding per-plan section (`## Plan <ID>: ...`) that contains a Goal subsection, a Briefs table, and a Briefs detail subsection.

**Hard reject 3 — flat structure (skip if multi-plan):** If no `## Dispatch order` table is present: at least one brief exists at the top level of the file.

**Hard reject 4 — dispatch graph:** Every "Depends on" entry in the dispatch table references a plan ID that exists in that same table, or is `—`. No cycles.

**Hard reject 5 — briefs have minimum content:** Every brief in every plan (or at the top level for flat ops) has at minimum a Goal section and an Acceptance criteria section.

If any hard reject fires, STOP. Print all failures together:

```
Plan validation failed:
  - [check 1] First H1 is not "# Operation: <slug>" — found: "# My Operation"
  - [check 5] Brief A.02 has no Acceptance criteria section

Fix these in the plan file and re-run /op-scaffold.
```

Do not proceed until all 5 pass.

## Step 2 — create Plane ticket hierarchy

### 2a — require plane-config

Check that `.claude/plane-config.md` exists in the current project. If absent, STOP:

```
op-scaffold requires .claude/plane-config.md — not found.
Run /ticket-install to set up the Plane backend for this project.
```

### 2b — load config and resolve IDs

Read from `.claude/plane-config.md`:
- API URL (field: `API URL:`)
- API key (field: `API key:`)
- Workspace slug (field: `Workspace slug:` or `- Workspace slug:`)
- Project ID (field: `Project ID:`)

Read from `.claude/ticket-config.md`:
- Project identifier (e.g. `CCONF`) — used only for display

Call `mcp__plane__list_states` to confirm the Backlog state ID. Cross-reference with the ID in `plane-config.md` (field `- Backlog:`). Use the live result.

Call `mcp__plane__list_labels` to confirm the `plan-ticket` label ID. Cross-reference with the ID in `plane-config.md`. Use the live result.

### 2c — create the operation ticket

Parse the slug from the first H1 (`# Operation: <slug>`). Title-case the slug for the ticket title (e.g. `my-op` → "My Op"; `auth-rewrite` → "Auth Rewrite").

Create one operation ticket:
- Title: `<title-cased slug>` (e.g. "Auth Rewrite")
- Labels: [`plan-ticket`]
- State: Backlog
- Parent: none
- Description: the "Why this exists" section text

Use `mcp__plane__create_work_item`. Record the returned sequence ID (e.g. `CCONF-42`) and internal UUID.

### 2d — create plan tickets (multi-plan only)

For each row in the `## Dispatch order` table, create one plan ticket as a child of the operation ticket:
- Title: `Plan <ID>: <Name>` (e.g. "Plan A: Auth State Refactor")
- Labels: [`plan-ticket`]
- State: Backlog
- Parent: operation ticket UUID
- `description_html`: the plan's Goal section text (rendered as HTML)

Use `mcp__plane__create_work_item`. Record each plan ticket's sequence ID and UUID.

### 2e — create brief tickets

For each brief in each plan (or at the top level for flat ops), create one brief ticket:
- Title: `<plan-id>.<brief-number>: <brief-slug>` for multi-plan (e.g. "A.01: extract-state-model"); just `<brief-number>: <brief-slug>` for flat ops
- Labels: none (brief tickets are not `plan-ticket`)
- State: Backlog
- Parent: the plan ticket UUID (multi-plan), or the operation ticket UUID (flat)
- `description_html`: rendered HTML of all available sections — Goal, Inputs, Outputs, Acceptance criteria, Notes, Out of scope

Use `mcp__plane__create_work_item`. Record each brief ticket's sequence ID and UUID.

### 2f — wire blocked_by relations

Wire dependencies from the plan doc's "Depends on" columns.

**Plan-level deps (multi-plan):** For each row in the dispatch table that has a non-`—` "Depends on" entry, create a `blocked_by` relation: the current plan ticket is blocked_by each plan ticket it depends on.

**Brief-level deps:** For each brief that has a non-`—` "Depends on" entry in its plan's briefs table, create a `blocked_by` relation: the current brief ticket is blocked_by each brief ticket it depends on (within the same plan).

Use `mcp__plane__create_work_item_relation` with relation type `blocked_by`.

### 2g — post summary comment on operation ticket

Post a comment on the operation ticket (use `mcp__plane__create_work_item_comment`) with:
- All created ticket IDs (operation, plans, briefs) in a readable list
- The ready-to-run `/tch` command (see Step 3 format)

## Step 3 — present summary

After all tickets are created, print:

```
Operation scaffolded: <slug>

Operation ticket: <PROJ-NN>
Plans:
  Plan A: <PROJ-NN> — "<Plan Name>" (N briefs)
  Plan B: <PROJ-NN> — "<Plan Name>" (N briefs)
  ...

Brief tickets:
  A.01: <PROJ-NN> — <brief-slug>
  A.02: <PROJ-NN> — <brief-slug>  [blocked_by A.01]
  ...

Ready to run:
  /tch <PROJ-NN>
```

Where `<PROJ-NN>` in the `/tch` command is the **operation ticket** sequence ID. The ticket-chain command drives work from the top-level ticket down through the hierarchy.

## Flat single-plan operations

If the plan has no `## Dispatch order` section, scaffold flat — operation ticket + brief tickets as direct children (no plan intermediary):

- Operation ticket: title-cased slug, `plan-ticket` label, Backlog
- Brief tickets: children of the operation ticket, no `plan-ticket` label, Backlog
- Dependency relations between brief tickets as specified in the briefs table

The `/tch` command is the same: `/tch <operation-ticket-id>`.

## Things you must not do

- Don't write any files to disk. No `docs/operations/` folders, no JSON state files, nothing.
- Don't run the operation. Ticket creation only. The user launches via `/tch`.
- Don't auto-trigger `/tch`. Always wait for explicit human go.
- Don't proceed if `.claude/plane-config.md` is absent — error, not skip.
- Don't skip hard-reject validation. There is no "scaffold anyway" path.
