# Scaffold an Operation From a Plan

You are scaffolding a multi-plan operation from a master plan file. The user will give you a path to a plan markdown file (typically generated by an external Opus using `~/.claude/operation-templates/META_PROMPT_FOR_PLAN_OPUS.md`, or hand-written following the format spec inlined in Step 1 below).

Your job: parse the plan, validate it, expand it into a runnable folder structure under `docs/operations/<slug>/`, and present the result for human review. **You do not run the operation.** That's `/op-run`.

## Input

The argument is a path to a plan markdown file. Examples:
- `/op-scaffold docs/proposals/<operation-slug>-plan.md`
- `/op-scaffold ~/Downloads/auth-rewrite-plan.md`

If no argument is given, ask the user for the plan file path.

## Step 1 — read and validate

1. Read the plan file end-to-end.
2. Verify it conforms to the plan format spec below. If validation fails, STOP and print specific gaps; do not create any files.

### Plan format spec (canonical)

The plan file must conform to this structure. The scaffolder rejects anything else.

#### Top of file (required)

```markdown
# Operation: <slug>

<one-paragraph framing — what this operation does, why now>

## Why this exists

<1–4 paragraphs of context: the diagnosis, the cost of not doing it,
what the world looks like after.>
```

- `# Operation: <slug>` — the slug is parsed from this line. Rule: lowercase, kebab-case, no spaces, ≤40 chars.
- `## Why this exists` — required H2 section.

#### Multi-plan path (preferred)

A plan with multiple work plans must include:

```markdown
## Dispatch order

| Plan | Name | Depends on | Effort | Parallel-safe with |
|------|------|------------|--------|---------------------|
| A    | <name> | — | N-M days | — |
| A2   | <name> | A complete | N-M days | — |
| B    | <name> | A2 complete | N-M days | — |
```

- **Plan IDs**: short — A, A2, B, C, D. One letter + optional digit.
- **Depends on**: must reference plan IDs that appear in the same table, or `—`. The scaffolder topologically sorts; cycles are an error.
- **Parallel-safe with**: optional, comma-separated list of plan IDs this plan can run alongside. Empty/`—` means "must be sequential against everything not yet complete."

For each row in the dispatch table, a per-plan section is required:

```markdown
## Plan A: <Plan Name>

### Goal

<one paragraph — what this plan delivers, what changes after>

### Briefs

| # | Slug | One-line intent | Depends on | Files touched (rough) |
|---|------|-----------------|------------|----------------------|
| 01 | <kebab-slug> | <intent> | — | <paths> |
| 02 | <kebab-slug> | <intent> | 01 | <paths> |

### Briefs — detail

#### Brief 01: <slug>

**Goal.** <1–2 sentences>

**Inputs.** Files to read (paths), reference docs (paths), context the worker needs.

**Outputs.** Files to edit/create (paths), what each produces.

**Acceptance criteria.**
- [ ] <concrete checkable thing>
- [ ] <test/grep that must pass>

**Verification.** Bash commands the task-lead runs to confirm. Each command should have an expected result inline. **All commands must be runnable as written** — no `<placeholder>`, no `# replace this with...`, no example slugs. The scaffolder rejects briefs containing placeholder markers.

The scaffolder PREPENDS the operation-level command set (from `## What done looks like`) to every brief's Verification block at expansion time. Brief authors should NOT repeat the op-level commands here — list only brief-specific checks. After scaffolding, every brief's Verification block reads:

```bash
# (auto-prepended) operation-level command set
<every executable command from 00-master.md's "What done looks like">

# brief-specific checks
rg '<symbol>' src/    # → ≥1 hit
```

**Notes / gotchas.** Anything subtle the worker needs to know.

**Out of scope.** What this brief does NOT do.
```

- "Depends on" in the briefs table lists brief IDs within THIS plan only.
- Slug rule: lowercase, kebab-case, ≤30 chars.

#### Acceptance-criteria authoring rules

These rules apply to every brief's `**Acceptance criteria.**` list. The scaffolder validates them at expansion time.

1. **Drive the real path, not a stand-in.** If a criterion is "function/feature X works," the verifying test must invoke X with its real signature against the real implementation surface. Tests that drive a stub, mock, fake, or in-memory adapter the brief itself controls do NOT satisfy the criterion. If the test surface unavoidably stubs the function under test, the brief MUST include a separate verification command that exercises the real path (a build, a project-aware type-check, an integration test, or equivalent).

2. **Behavior tests for code-deletion work.** If a brief removes a code path, retires a feature, or deletes a function/module, acceptance MUST include at least one behavior test that drives a representative input through the system and asserts the deleted behavior does not manifest. Symbol-grep alone is insufficient. (Failure mode: the symbol disappears from the source but the behavior still fires because instructions, prompts, or callers in adjacent files preserved it.)

3. **Automated UI verification is mandatory.** If a brief produces or modifies user-visible UI — any rendered output a human will see (DOM, canvas, CLI output formatting, log structure, generated docs, terminal UI, etc.) — its acceptance criteria MUST include at least one automated verification that exercises the rendered surface: integration test, snapshot/visual-regression test, screenshot diff, DOM assertion in a headless browser, or equivalent. If the project genuinely cannot support automated UI verification at the time of the brief (e.g., no UI test harness exists yet), the brief MUST cite a tracker ticket whose disposition is "add UI testing for this surface" AND fall back to a manual smoke step in acceptance. The ticket reference is the justification; "I didn't feel like writing a test" is not.

4. **Cross-brief invariant ownership.** When multiple briefs touch the same file, each brief's acceptance describes only its own contribution. File-level invariants ("no remaining imports of Y", "no callers of removed function Z", "the pipeline emits exactly N stages") belong on the LAST brief to touch that file in the dispatch order, OR on the operation's `## What done looks like` if no single brief is clearly last. The scaffolder warns when a file appears in two or more briefs' "Files touched" entries — confirm cross-brief invariants are placed correctly before proceeding.

#### Flat single-plan path

If the plan has no `## Dispatch order` section, scaffold flat — briefs at the top level, no per-plan section.

#### "What done looks like" (required, at end of file)

```markdown
## What done looks like

When all plans report complete:

```bash
npm run build       # must pass — this is the authoritative typecheck path
npm test            # must pass
rg '<symbol>' src/  # → expected hits/no hits
```

Plus narrative integration checks (optional, in prose):

1. <integration check — e.g., live smoke test, dogfood loop>
2. <any cleanup/dead-code criteria>
```

This block has TWO required parts:

1. **An executable command set** in a `bash` fenced block. These are the gate. The scaffolder PREPENDS this block verbatim to every brief's Verification block, so what passes here passes everywhere. Pick the authoritative paths — if the project has both `tsc --noEmit` (root convenience) and `tsc -b` (project-aware via build), use `npm run build` (or whatever script invokes `tsc -b`). The build is the lowest common denominator, NOT typecheck. A brief that passes `tsc --noEmit` but fails `npm run build` will not catch the failure until operation-level verification at the end.

2. **Narrative integration checks** (optional). For things a human reads in HANDOFF.md or executes once at the end. Not gating, not auto-prepended.

The scaffolder rejects plans whose "What done looks like" has no executable commands — prose-only is insufficient because there is nothing to prepend to brief verification blocks.

This is what the **operation-level conductor** uses to verify the whole operation, not just individual plans. The same commands run at brief, plan, and operation level — the LCD propagates.

### Validation checklist

Verify in order. Each check is either a **HARD REJECT** or a **SOFT WARNING**. The two have different semantics — do not blur them:

- **HARD REJECT**: refuse to scaffold; print the gap; exit. Do NOT offer a "scaffold anyway" path. Do NOT ask the user to choose between strict and lenient. Do NOT frame the gap as a "judgment call" or surface it as a question. The plan is wrong; the human fixes the plan and re-runs `/op-scaffold`. The whole point of hard rejects is to remove plan-author/scaffolder discretion at scaffold time — re-introducing it via "want me to proceed anyway?" defeats the gate.
- **SOFT WARNING**: print, ask the human to confirm before proceeding. Soft warnings are for things the scaffolder cannot judge mechanically (e.g., is the cross-brief invariant on the right brief?) — they exist precisely because the scaffolder shouldn't auto-decide.

**Hard rejects:**

1. **Top section**: First H1 line is `# Operation: <slug>` (kebab-case slug, ≤40 chars). Has a `## Why this exists` section.
2. **Multi-plan**: has a `## Dispatch order` table with the five required columns. Has a per-plan section for every row in the table. Every per-plan section has Goal, Briefs (table), and Briefs — detail.
3. **Flat single-plan**: no `## Dispatch order` section, just briefs at the top level.
4. Has a `## What done looks like` section at the bottom.
5. **Dispatch graph**: every "Depends on" entry references a plan ID that exists in the table; no cycles.
6. **Per-brief**: every brief has Goal / Inputs / Outputs / Acceptance criteria / Verification.
7. **Op-level commands present**: `## What done looks like` contains at least one `bash` fenced code block with at least one executable command. Prose-only is rejected — the scaffolder needs commands to prepend to brief verification blocks.
8. **No placeholder verification commands**: scan every brief's Verification block AND the op-level `## What done looks like` bash block. Reject if any command line contains `<...>` (angle-bracket placeholders), `# replace`, `# TODO`, `<your-`, `<path-to-`, or other obvious placeholder markers. List each offending brief + line.
9. **UI briefs have automated verification or escape hatch**: a brief is treated as UI-touching if its Goal, Outputs, or "Files touched" mention any of: `.tsx`, `.jsx`, `.vue`, `.svelte`, `.css`, `.scss`, `.html`, webview, canvas, render, kanban, sidebar, panel, dashboard, dialog, modal, terminal output, log format, generated docs, CLI output. (Heuristic — when in doubt, treat as UI.) Such a brief MUST satisfy ONE of:
   - **Acceptance criteria** (the `**Acceptance criteria.**` list, NOT the Verification block, NOT the Notes section) includes at least one item that explicitly names an automated verification of the rendered surface (integration test, snapshot test, visual-regression test, screenshot diff, headless-browser DOM assertion). The item must name the test file/path or assertion specifically — generic items like "tests pass" or "frontend tests are updated" do NOT satisfy the rule.
   - **Acceptance criteria** includes a manual smoke step AND cites a tracker ticket ID (matching the project's ticket-ID regex from CLAUDE.md or `.claude/`) for adding automated coverage. The literal text "TODO add UI tests" without a ticket reference is rejected.

   This is a **HARD REJECT — no judgment calls**. Verification block items, Notes-section mentions, implied-by-other-criteria reasoning, and "the existing test surely covers this" arguments do NOT satisfy the rule. The acceptance list is the contract the worker is measured against; if the UI verification isn't there, the worker can ship without it. The fix is a 1-line acceptance edit by the plan author, then re-run `/op-scaffold` — do NOT offer to scaffold-anyway, do NOT ask the user to choose between strict and lenient. On reject, print the brief ID, quote the offending acceptance list verbatim, and exit with the other hard rejects.

**Soft warnings (proceed only after human confirms):**

10. **Cross-brief file overlap**: if a file path appears in two or more briefs' "Files touched" entries, print:
    ```
    ⚠ File <path> appears in briefs: A.01, A.03, B.02
       Confirm cross-brief invariants ("no remaining imports of X", etc.) live on
       the LAST brief to touch this file (B.02), not the first. See the
       acceptance-criteria authoring rules in op-scaffold.md.
    ```
11. **Deletion brief lacks behavior test**: if a brief's title, goal, or one-line intent matches `/\b(remove|delete|rip out|clean ?up|retire|drop|eliminate)\b/i`, scan its acceptance criteria. Warn if every acceptance line is a grep / symbol-existence check (matches `/^- \[ \] (rg |grep |no (more )?(imports?|callers?|references?) of)/`). Print:
    ```
    ⚠ Brief <ID> appears to remove a code path. Acceptance criteria are grep-only.
       Per the acceptance-criteria authoring rules, deletion briefs MUST include
       at least one behavior test that drives a representative input through the
       system and asserts the deleted behavior does not manifest. Symbol-grep
       alone is insufficient.
    ```

For soft warnings, after listing all of them, prompt: `Proceed despite warnings? (yes / no)`. On `no`, exit without scaffolding.

If any hard reject fires, STOP. Print all hard rejects together (don't fail-fast on the first one — surface everything that's wrong so the human can fix in one pass):

```
Plan validation failed (hard rejects):
  - Missing section: ## Why this exists
  - Plan B depends on plan X, which is not in the dispatch table
  - Brief A.03 has no "Verification" section
  - "What done looks like" has no executable bash block (prose only)
  - Brief A.04 verification command contains placeholder: `rg '<symbol>' src/`
  - Brief B.01 touches kanban.tsx but has no automated UI verification and no ticket reference

Fix these in the plan file and re-run /op-scaffold.
```

Do not create any files until all hard rejects are resolved AND the human has confirmed any soft warnings.

## Step 2 — expand to disk

Create the operation folder and all derived files:

```
docs/operations/<slug>/
  00-master.md             # the original plan, copied verbatim
  README.md                # auto-generated: overview + dispatch graph + status table
  HANDOFF.md               # placeholder, conductor fills at end
  VERIFY.md                # placeholder, conductor fills at end (manual-eyeball checklist)
  operation-state.json     # initial state (all plans pending)
  plan-<ID>-<plan-slug>/
    00-conductor.md        # auto-generated from per-plan section + briefs
    01-<brief-slug>.md     # auto-generated from per-brief detail (Verification block prepends op-level command set)
    02-<brief-slug>.md
    ...
  (one folder per plan)
```

Plan slug rule: take the plan name from the dispatch table, lowercase, kebab-case, ≤30 chars. So `Plan A: Canonical Interview State (CIS)` → `plan-A-canonical-interview-state`.

Brief slug rule: take the brief slug from the per-plan briefs table.

### `00-conductor.md` template (per plan)

```markdown
# Plan <ID> conductor brief — <Plan Name>

**You are operation-task-lead.** You have been dispatched by operation-conductor to drive plan <ID> ("<Plan Name>") of the "<operation slug>" operation.

## Plan goal

<copy from the plan's ### Goal section>

## Briefs in this plan

| # | Slug | One-line intent | Depends on |
|---|------|-----------------|------------|
<copy from the plan's ### Briefs table — Files-touched column omitted from this view>

## Brief dependency graph

<auto-generated ASCII graph from the "Depends on" column>

Example for an A → B,C → D shape:
```
01 ─┬─> 02 ──┐
    └─> 03 ──┴─> 04
```

## Parallel dispatch policy

- Briefs with no unsatisfied deps and no overlapping output files MAY be dispatched in parallel.
- The brief table's "Files touched" column lists overlap risks. Briefs that touch the same file must be serialized.
- The following parallel batches are safe (auto-computed):
  * Batch 1: <brief IDs>
  * Batch 2: <brief IDs>
  * ...

## Plan-level verification — what "plan done" means

```bash
# (auto-prepended) operation-level command set from 00-master.md's "What done looks like"
<every executable command from the master plan's "What done looks like" bash block, verbatim>

# plan-specific checks
git log --oneline <baseline>..HEAD | grep "<operation-slug> \[plan-<ID>\]"   # expected commit count
<any plan-specific greps/tests from the plan's narrative>
```

The op-level command set is the gate. It is the same set every brief in this plan ran during their own verification — running it again at plan level catches integration breakage between briefs.

## Constraints

- Branch: <current branch name from `git branch --show-current`>
- Commit prefix: <derive from operation slug — e.g. `<operation-slug> plan-<ID>` or `<TKT>-NNN plan-<ID>`>
- Rework cap: 3 attempts per brief.
- Workers commit nothing — you commit after verification.
- Do not modify briefs.
- Refer to `~/.claude/agents/operation-task-lead.md` for the full standard task-lead procedure (dispatch / verify / rework / commit / report).
```

### Numbered brief files (per brief)

Copy the per-brief detail from the master plan verbatim. Standard sections preserved: Goal, Inputs, Outputs, Acceptance criteria, Verification, Notes / gotchas, Out of scope.

**Verification block expansion:** when copying the brief's Verification block into the numbered file, the scaffolder PREPENDS the operation-level command set (the bash block from `00-master.md`'s `## What done looks like`) to the brief's own Verification commands, with a separator comment. So the scaffolded brief's Verification section reads:

````markdown
**Verification.**

```bash
# (auto-prepended) operation-level command set — the gate
<every executable command from 00-master.md's "What done looks like", verbatim>

# brief-specific checks (from the master plan)
<the brief's own Verification commands, verbatim>
```
````

The worker runs this whole block in order, top to bottom. The op-level commands are the gate at every level — brief, plan, operation. A brief that passes its own checks but breaks the op-level set fails verification at the brief, not three days later.

Add a header line at the top of the file:

```markdown
# Brief NN: <slug>

**Plan:** <ID> — <Plan Name>
**Depends on:** <from briefs table>
**Files touched:** <from briefs table>

## Goal
...
```

## How this gets run

This operation is driven by the `/op-run` slash command. Run it with:

```
/op-run docs/operations/<slug>/
```

The main `/op-run` session plays conductor + task-lead inline (the Claude Code harness strips the `Agent` tool from spawned subagents, so the original three-tier dispatch chain doesn't run as subagents — see `~/.claude/agents/operation-conductor.md` and `~/.claude/agents/operation-task-lead.md` for the canonical procedures the main session inlines). The session dispatches `operation-worker` (Sonnet) subagents per brief, parallelized within each plan per the brief dependency graph, and sends a Prowl notification when the whole operation is complete (or blocked).

Progress is tracked in [operation-state.json](operation-state.json). The final report lands in [HANDOFF.md](HANDOFF.md) when the operation completes.

```json
{
  "operation": "<slug>",
  "started_at": null,
  "branch": "<current branch>",
  "plans": {
    "<ID>": {
      "status": "pending",
      "attempts": 0,
      "task_lead_summary": null,
      "started_at": null,
      "completed_at": null,
      "briefs": {
        "01": { "status": "pending", "residuals": [] },
        "02": { "status": "pending", "residuals": [] }
      }
    }
  },
  "rework_rounds": 0,
  "final_verification": null,
  "blocker": null,
  "completed_at": null

```

The `residuals` array on each brief is filled in by the task-lead during the disposition step (see `~/.claude/agents/operation-task-lead.md` step 4a). Each entry has shape `{ description, disposition, ticket_id_or_brief_id?, justification? }` where `disposition` is one of `fold-in`, `follow-up`, or `accept-with-justification`. A residual missing the `disposition` field blocks operation completion.

## Flat single-plan operations

If the plan has no `## Dispatch order` section, scaffold flat:

```
docs/operations/<slug>/
  00-master.md
  00-conductor.md       # treats the whole operation as one plan
  README.md
  HANDOFF.md
  VERIFY.md
  operation-state.json
  01-<brief>.md         # Verification block prepends op-level command set
  02-<brief>.md
  ...
```

The `operation-state.json` for flat operations has a single plan key, with the same `briefs` map shape (each brief has a `residuals` array):
```json
{ "plans": { "main": { "status": "pending", "briefs": { "01": { "status": "pending", "residuals": [] } }, ... } } }
```

`/op-run` handles flat layouts by treating the operation root as a single "main" plan.

## Things you must not do

- Don't run the operation. Scaffolding only. The user reviews before launching.
- Don't modify the master plan content while copying it into `00-master.md` — verbatim copy.
- Don't create files outside `docs/operations/<slug>/`.
- Don't auto-trigger `/op-run`. Always wait for explicit human go.
