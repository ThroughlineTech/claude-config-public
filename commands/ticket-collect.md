# Collect Work From Delegated Tickets

After another agent (e.g. Gemini in Copilot) executes briefs generated by `/ticket-delegate`, this command verifies the work, reviews it (for full-lifecycle delegations), and transitions tickets to the next status.

**Supports multiple IDs** for batch collection: `/ticket-collect 10 11 12 13`. Reviews each ticket and generates a consolidated `CHAIN-REVIEW-*.md` checklist (same format as `/ticket-chain`).

## Input
Argument: `{ID} [...]` — one or more ticket IDs (e.g. `TKT-005` or `10 11 12 13`)

**ID shorthand:** If an argument is a bare number (e.g., `5` or `26`), resolve it to a full ticket ID: read the ticket prefix from `.claude/ticket-config.md`, scan existing ticket files to determine the zero-padding width, and expand (e.g., `5` → `TKT-005`).

## Pre-flight Checks
- `.claude/ticket-config.md` must exist
- The ticket file must exist at `{tickets-dir}/{ID}.md`. If it lives in a terminal subfolder (`shipped/`, `deferred/`, `wontfix/`), STOP — collect only operates on active tickets.
- Ticket status MUST be `delegated`. If not, report status and stop.
- The Delegation Log section must exist with at least one entry (otherwise nothing is delegated to collect)

## Steps

1. **Read the ticket file** including the Delegation Log. The most recent entry tells you which phase was last delegated.

2. **Find the brief file** for that phase (`{tickets-dir}/{ID}.{phase-tag}.brief.md`).

3. **Phase-specific verification** — confirm the executing agent actually wrote what the brief asked for:

   ### `full` (no phase — full lifecycle delegation)

   This is the most comprehensive collect. Claude acts as **code reviewer**, not just a checkbox verifier. The other model did the investigation and implementation — Claude reviews whether it meets the bar.

   **Structural checks (must all pass or leave at `delegated`):**
   - Investigation, Proposed Solution, and Implementation Plan sections are filled in
   - Feature branch has new commits beyond main (`git log {branch} ^main --oneline`)
   - Files Changed and Test Report sections are filled in

   **Substantive review (Claude reads and judges):**

   1. **Read the investigation.** Is the root cause / architectural fit analysis sound? Did the agent explore deeply enough or take shortcuts? Is the regression risk assessment reasonable?
   2. **Read the full diff:** `git diff main...{branch}`. Review the actual code changes:
      - Does the implementation match the acceptance criteria?
      - Are there bugs, security issues, or logic errors?
      - Does it follow project conventions (from `CLAUDE.md`)?
      - Are edge cases handled?
      - Is the code clean — no dead code, no debugging artifacts, no commented-out blocks?
   3. **Review the tests.** Are they testing real behavior or just smoke tests? Do they cover the acceptance criteria? Are there obvious missing test cases?
   4. **Check the Test Report.** Did all tests pass? Is the build clean?

   **Write a `## Delegation Review` section** into the ticket with:
   - **Verdict:** `approved` | `concerns` | `rejected`
   - **Summary:** 2-3 sentences on the overall quality
   - **Issues found:** numbered list (if any). Each issue: what's wrong, where (file:line), severity (blocking / should-fix / nit)
   - **Missing:** anything the acceptance criteria required that isn't in the diff

   **Transition logic:**
   - `approved` (no blocking issues): transition to `review`, update Files Changed + Test Report if the agent's versions are incomplete
   - `concerns` (non-blocking issues found): transition to `review` but flag the concerns prominently in the output. The user decides whether to fix before shipping.
   - `rejected` (blocking issues — tests fail, acceptance criteria not met, security problem): leave at `delegated`. Report what needs fixing. The user can re-delegate or fix manually.

   ### `investigate`
   - Verify the ticket file's Investigation, Proposed Solution, and Implementation Plan sections are now filled in (non-empty, not just placeholder text)
   - If they're empty: report "investigation didn't return content; brief may not have been executed" and leave status as `delegated`
   - If filled: transition status to `proposed`

   ### `implement`
   - Verify the feature branch has new commits beyond main (`git log {branch} ^main --oneline`)
   - Read the diff: `git diff main...{branch} --stat` and full `git diff main...{branch}`
   - Update the ticket's "Files Changed" section based on the diff (one line per file)
   - Update the ticket's "Test Report" section: extract any test counts the executor mentioned in commit messages, or note "test results not reported in commits — verify manually with /ticket-review"
   - Transition status to `review`

   ### `review`
   - Verify the ticket's "Verification Checklist (for human)" section is filled in
   - If filled: transition status to `review` (already there) and just confirm
   - If not: leave as `delegated`

   ### `verify investigate`, `verify implement`, `verify review`
   - Verify a "## Peer Review ({verify-target})" section exists in the ticket
   - If filled: transition status BACK to whatever it was before the verify (use the Delegation Log to determine: investigate→proposed, implement→review, review→review)
   - Print a summary of the peer review's key points so the user can decide whether to revise or proceed
   - Suggest next action based on what the peer review found:
     - If the review said "looks good": "no issues raised; ready to /ticket-{next-step}"
     - If the review raised concerns: "peer review raised concerns; consider re-running /ticket-{phase} with revisions"

4. **Append to the Delegation Log** a "collected" entry: timestamp, phase, status transition.

5. **Update the `updated` date** in the ticket frontmatter.

6. **Output a summary**:
   ```
   {ID} collected from {phase}

   Status: {old-status} → {new-status}

   {phase-specific summary, e.g. "3 files changed, 2 commits, 47 tests added"
    or "peer review raised 2 concerns: ..."}

   Next: {suggested next command, e.g. "/ticket-review {ID}" or "/ticket-investigate {ID} --revise"}
   ```

## Batch Collection (multiple IDs)

When multiple IDs are given, process each ticket through the single-ticket steps above, then generate a consolidated output:

### Step 1: Review each ticket

Run the phase-specific verification (above) for each ticket. Track:
- Which tickets were `approved`
- Which had `concerns` (non-blocking)
- Which were `rejected` (blocking)
- Which had structural issues (brief not executed, no commits, etc.)

### Step 2: Copy ticket files from worktrees (if parallel delegation was used)

For each ticket with a worktree at `.worktrees/ticket-{lowercased-id}/`:
- Copy the updated ticket file back to `{tickets-dir}/{ID}.md`
- The diff and commits remain on the worktree's branch

### Step 3: Generate consolidated review checklist

If any tickets were `approved` or had `concerns`, generate `{tickets-dir}/CHAIN-REVIEW-{YYYY-MM-DD-HHMM}.md` — same format as `/ticket-chain` produces. One checklist with verification steps per ticket, including:
- What changed per ticket
- Acceptance criteria verification steps
- Edge case and regression checks
- Per-ticket verdict checkboxes (pass / fail with reason) — inline after each ticket and rolled up in a Results section at the bottom

Deploy to preview/staging if configured (same as `/ticket-chain` Phase 4).

### Step 4: Output summary

```
{N} tickets collected

Approved:  {n} (ready to ship)
Concerns:  {c} (non-blocking issues flagged — review before shipping)
Rejected:  {r} (blocking issues — needs rework)
Not ready: {x} (brief not executed or incomplete)

Review checklist: {path to CHAIN-REVIEW-*.md}

Approved:
  TKT-010  "title"  approved (2 nits)
  TKT-012  "title"  approved

Concerns:
  TKT-011  "title"  concerns: missing error handling in upload.ts

Rejected:
  TKT-013  "title"  rejected: tests failing, 2 acceptance criteria not met

Next:
  Review the checklist:  open {checklist path}
  Ship approved tickets: /tsh 10  (or /tsh 10 12 when batch ship is supported)
  Re-delegate rejected:  /ticket-delegate 13
  Clean up worktrees:    /tcl --all
```

Prowl once:
- Application: `Claude Code: {project-name}`
- Event: `Delegation collected — {n} approved, {r} rejected`
- Description: ticket IDs and verdicts
- Priority: `1` if any rejected, `0` otherwise

## Rules
- Do NOT modify code during collect. This is read-only on the codebase (it does write to the ticket file).
- Do NOT delete the brief file — keep it as part of the ticket's audit trail.
- If the executing agent's work is incomplete (sections not filled, no commits, etc.), DO NOT force a status transition. Report the gap and leave status as `delegated` so the user can re-delegate or finish manually.
- If the work was implementation but the agent committed to the wrong branch (not the feature branch), report it and stop — don't try to fix it automatically.
