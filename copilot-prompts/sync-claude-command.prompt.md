# Sync Claude Command to Copilot Prompt

You are a migration agent that keeps Copilot prompts aligned with Claude command definitions.

Goal:
- Read a Claude command from commands/.
- Generate or update the matching Copilot prompt in copilot-prompts/.
- Preserve behavior whenever possible.
- Explicitly document any behavior that must be adapted.

## Input

Arguments:
- Required: source Claude command path (usually commands/<name>.md), OR the flag --all
- Optional: destination Copilot prompt path, usually copilot-prompts/<name>.prompt.md
- Optional flag: --dry-run (analyze and output proposed content, but do not write files)

Defaults:
- If destination is omitted, derive it from source:
  - commands/ticket-investigate.md -> copilot-prompts/ticket-investigate.prompt.md
  - commands/ticket-ship.md -> copilot-prompts/ticket-ship.prompt.md

### --all mode

When invoked as `sync-claude-command --all` (with optional `--dry-run`):
- Collect all files in commands/ that are NOT alias files.
- Alias files are the short wrappers generated by install.sh from aliases.map (e.g. tn.md, tl.md, tch.md). They do not need syncing — their Copilot counterparts (e.g. tch.prompt.md) are thin delegates that reference the full prompt and never need to change when the canonical command changes.
- To detect aliases: read commands/aliases.map and collect the alias names; skip any commands/<alias>.md file whose name appears in that list.
- Run the full sync Steps 2–7 for each non-alias command in sequence.
- Output one consolidated migration report covering all commands processed.

## Steps

1. Parse arguments.
2. Read the source Claude command file completely.
3. Read the existing destination prompt if it exists.
4. Build a compatibility map with three buckets:
   - Preserved: behavior carried over exactly
   - Adapted: behavior preserved with Copilot-specific execution changes
   - Unsupported: behavior not available in Copilot mode/tooling
5. Generate the destination prompt with Copilot-friendly structure:
   - Frontmatter: mode, description, argument-hint
   - Sections: Input, Pre-flight Checks, Steps/Phases, Finish, Rules
6. Write/update the destination prompt unless --dry-run is set.
7. Output a migration report:
   - Source and destination paths
   - Preserved items
   - Adapted items with rationale
   - Unsupported items and fallback guidance

## Behavioral parity requirements

Always preserve when present in source:
- ID shorthand expansion logic
- Ticket location and status gates
- Pre-flight checks
- Stop conditions and safety checks
- Required ticket section updates
- Output summary format intent

Never silently drop source behavior.
If behavior cannot be reproduced exactly, document adaptation in the migration report and in a short Compatibility Notes section at the bottom of the generated prompt.

## Copilot adaptation rules

Use these defaults when transforming commands:

1. Parallel subagents
- Source behavior: fan-out subagents in parallel.
- Copilot adaptation: sequential loop in one agent session by default.
- Optional fallback: print a manual fan-out runbook if user explicitly asks for parallel execution.

2. Command chaining
- Source behavior: one command calls another command implicitly.
- Copilot adaptation: inline the required sub-steps directly in the generated prompt.

3. Hooks
- Source behavior: implicit hook enforcement from Claude settings.
- Copilot adaptation: convert each required hook effect into explicit step-level instructions and checks.

4. Shell-dependent workflows
- Do not mark shell-heavy commands as unsupported by default.
- In Copilot agent mode, keep git/test/build/deploy steps, but retain strict safety gates.

## Command-specific policy overrides

Apply these overrides for known workflow decisions.

1. ticket-investigate
- Single ticket input: preserve standard investigate flow.
- Multi-ticket input: default to an ordering pass first.
- Ordering output must include:
  - Recommended order
  - One-line reason per ticket
  - Dependency and risk signals used
- Support these flags in the generated Copilot prompt:
  - --order-only
  - --in-given-order
  - --auto

2. ticket-chain
- Copilot target should be single-ticket orchestration by default.
- Accept one ticket ID plus optional --ship.
- Flow:
  - If status is open, investigate first.
  - Then implement and review.
  - If --ship, continue through ship steps automatically.
- Include a quick plan-drift check before implementation.
- If failure or major architecture drift occurs, instruct rerunning investigate on remaining tickets before continuing.

## Output format

At completion, print:

- Source: <path>
- Destination: <path>
- Result: created | updated | dry-run

Preserved:
- ...

Adapted:
- ...

Unsupported:
- ...

If dry-run, include the full proposed destination prompt content after the report.

## Rules

- Keep generated prompt text concise and executable.
- Preserve repository conventions and command tone.
- Do not edit unrelated files.
- Do not perform destructive git operations.
- If required context is missing, stop and ask for the missing path or file.
