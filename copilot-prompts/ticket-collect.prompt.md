# Collect Work From Delegated Tickets

After another agent (e.g. Gemini in Copilot) executes briefs generated by `/ticket-delegate`, this command verifies the work, reviews it (for full-lifecycle delegations), and transitions tickets to the next status.

**Supports multiple IDs** for batch collection. Reviews each ticket and generates a consolidated `CHAIN-REVIEW-*.md` checklist.

## Input

Argument: `{ID} [...]` — one or more ticket IDs.

**ID shorthand:** If an argument is a bare number (e.g., `5` or `26`), resolve it to a full ticket ID: read the ticket prefix from `.claude/ticket-config.md`, scan existing ticket files to determine zero-padding width, and expand.

## Pre-flight Checks

- `.claude/ticket-config.md` must exist.
- The ticket file must exist at `{tickets-dir}/{ID}.md`. If in a terminal subfolder, STOP — collect only operates on active tickets.
- Ticket status MUST be `delegated`. If not, report status and stop.
- The Delegation Log section must exist with at least one entry.

## Steps (single ticket)

1. **Read the ticket file** including the Delegation Log. The most recent entry tells you which phase was last delegated.

2. **Find the brief file** for that phase (`{tickets-dir}/{ID}.{phase-tag}.brief.md`).

3. **Phase-specific verification:**

### `full` (full lifecycle delegation)

Claude acts as **code reviewer**, not just a checkbox verifier.

**Structural checks (must all pass or leave at `delegated`):**
- Investigation, Proposed Solution, and Implementation Plan sections are filled in.
- Feature branch has new commits beyond main: `git log {branch} ^main --oneline`.
- Files Changed and Test Report sections are filled in.

**Substantive review (Claude reads and judges):**
1. Read the investigation. Is the root cause / architectural fit analysis sound? Did the agent explore deeply enough?
2. Read the full diff: `git diff main...{branch}`. Review actual code changes:
   - Does the implementation match the acceptance criteria?
   - Are there bugs, security issues, or logic errors?
   - Does it follow project conventions from `CLAUDE.md`?
   - Are edge cases handled? Is the code clean?
3. Review the tests. Are they testing real behavior? Do they cover the acceptance criteria?
4. Check the Test Report. Did all tests pass? Is the build clean?

**Write a `## Delegation Review` section** into the ticket with:
- **Verdict:** `approved` | `concerns` | `rejected`
- **Summary:** 2–3 sentences on overall quality
- **Issues found:** numbered list (if any). Each: what's wrong, where (file:line), severity (blocking / should-fix / nit)
- **Missing:** anything the acceptance criteria required that isn't in the diff

**Transition logic:**
- `approved` (no blocking issues): transition to `review`
- `concerns` (non-blocking issues): transition to `review`, flag concerns prominently
- `rejected` (blocking issues): leave at `delegated`; report what needs fixing

### `investigate`

- Verify Investigation, Proposed Solution, and Implementation Plan sections are now filled in (non-empty, not placeholder text).
- If empty: report "investigation didn't return content; brief may not have been executed" and leave as `delegated`.
- If filled: transition status to `proposed`.

### `implement`

- Verify the feature branch has new commits beyond main: `git log {branch} ^main --oneline`.
- Read the diff: `git diff main...{branch} --stat` and full `git diff main...{branch}`.
- Update the ticket's "Files Changed" section based on the diff.
- Update the ticket's "Test Report" section with any test counts found in commit messages.
- Transition status to `review`.

### `review`

- Verify the ticket's "Verification Checklist (for human)" section is filled in.
- If filled: confirm status is `review` (no change needed).
- If not: leave as `delegated`.

### `verify investigate`, `verify implement`, `verify review`

- Verify a "## Peer Review ({verify-target})" section exists in the ticket.
- If filled: transition status back to what it was before the verify (use the Delegation Log: investigate→proposed, implement→review, review→review).
- Print a summary of the peer review's key points.
- Suggest next action:
  - If "looks good": "no issues raised; ready to /ticket-{next-step}"
  - If concerns: "peer review raised concerns; consider re-running /ticket-{phase} with revisions"

4. **Append to the Delegation Log** a "collected" entry: timestamp, phase, status transition.

5. **Update the `updated` date** in the ticket frontmatter.

6. **Output a summary:**
   ```
   {ID} collected from {phase}

   Status: {old-status} → {new-status}

   {phase-specific summary, e.g. "3 files changed, 2 commits, 47 tests added"}

   Next: {suggested next command}
   ```

## Batch Collection (multiple IDs)

### Step 1: Review each ticket

Run phase-specific verification for each ticket. Track:
- Approved
- Concerns (non-blocking)
- Rejected (blocking)
- Structural issues (brief not executed, no commits, etc.)

Process tickets sequentially in one session.

### Step 2: Copy ticket files from worktrees (if parallel delegation was used)

For each ticket with a worktree at `.worktrees/ticket-{lowercased-id}/`:
- Copy the updated ticket file back to `{tickets-dir}/{ID}.md`.
- The diff and commits remain on the worktree's branch.

### Step 3: Generate consolidated review checklist

If any tickets were `approved` or had `concerns`, generate `{tickets-dir}/CHAIN-REVIEW-{YYYY-MM-DD-HHMM}.md` — same format as `/ticket-chain` produces. One checklist with verification steps per ticket, including:
- What changed per ticket
- Acceptance criteria verification steps
- Edge case and regression checks
- Per-ticket verdict checkboxes (pass/fail) — inline after each ticket and rolled up in a Results section at the bottom

Deploy to preview/staging if configured.

### Step 4: Output summary

```
{N} tickets collected

Approved:  {n} (ready to ship)
Concerns:  {c} (non-blocking issues flagged — review before shipping)
Rejected:  {r} (blocking issues — needs rework)
Not ready: {x} (brief not executed or incomplete)

Review checklist: {path to CHAIN-REVIEW-*.md}

Approved:
  TKT-010  "title"  approved (2 nits)
  TKT-012  "title"  approved

Concerns:
  TKT-011  "title"  concerns: missing error handling in upload.ts

Rejected:
  TKT-013  "title"  rejected: tests failing, 2 acceptance criteria not met

Next:
  Review the checklist:  open {checklist path}
  Ship approved tickets: /tsh 10
  Re-delegate rejected:  /ticket-delegate 13
  Clean up worktrees:    /tcl --all
```

Notify the user (Prowl if available, otherwise print prominently):
- Application: `ticket-collect`
- Event: `Delegation collected — {n} approved, {r} rejected`
- Description: ticket IDs and verdicts
- Priority: `1` if any rejected, `0` otherwise

## Rules

- Do NOT modify code during collect. This is read-only on the codebase (writes to ticket files only).
- Do NOT delete the brief file — keep it as part of the ticket's audit trail.
- If the executing agent's work is incomplete, do NOT force a status transition. Leave as `delegated` so the user can re-delegate or finish manually.
- If the agent committed to the wrong branch, report it and stop — don't try to fix it automatically.

## Compatibility Notes

- **Parallel batch review → sequential loop:** source intent supports parallel review but the behavior is sequential in practice (reading and writing one ticket file at a time). This is preserved exactly.
- **Prowl notification → conditional:** Prowl is used if available via global CLAUDE.md; otherwise the summary is printed prominently.
