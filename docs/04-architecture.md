## Symlinks vs. regenerated files

Two different mechanisms for getting Layer 1 content into Layer 2:

### Symlinks (for most things)

```
~/.claude/CLAUDE.md  →  ~/src/claude-config/CLAUDE.md
~/.claude/plan-mode.md  →  ~/src/claude-config/plan-mode.md
~/.claude/brainstorm-mode.md  →  ~/src/claude-config/brainstorm-mode.md
~/.claude/commands   →  ~/src/claude-config/commands
~/.claude/plans      →  ~/src/claude-config/plans
~/.claude/brief-templates →  ~/src/claude-config/brief-templates

~/Library/Application Support/Code/User/prompts/*.md →
    ~/src/claude-config/copilot-prompts/*.md  (one symlink per file; loop in install.sh)
```

Symlinks are **live**: edit the file in Layer 1 and the change is immediately visible through the Layer 2 path. No re-install needed.

### Regenerated files (for four things)

Four files are not symlinked — they're regenerated on every `install.sh` run:

1. **`~/.claude/settings.json`** — built by merging `settings.base.json` with the platform-specific file via `jq`. Can't be symlinked because it's a platform-specific merge result.

2. **`copilot-prompts/claude-global.instructions.md`** (in the repo) — built by prepending `---\napplyTo: "**"\n---\n\n` frontmatter to the content of `CLAUDE.md`. Can't be symlinked because it needs the frontmatter header that `CLAUDE.md` doesn't have. (The file is then symlinked into VS Code's user prompts directory, so the symlink chain is Layer 2 → Layer 1-generated → Layer 1-source.)

3. **`copilot-prompts/plan-mode.instructions.md`** (in the repo) — built the same way from `plan-mode.md`. Same symlink chain into VS Code's user prompts directory.

4. **`copilot-prompts/brainstorm-mode.instructions.md`** (in the repo) — built the same way from `brainstorm-mode.md`.

**Edit `CLAUDE.md`, `plan-mode.md`, or `brainstorm-mode.md` and you need to re-run `install.sh`** to regenerate the matching Copilot instructions file. Same for any `settings.*.json` edit. Edit anything else and it's live.

## The settings merge

`install.sh` runs this jq expression:

```bash
jq -s '
  .[0] as $base | .[1] as $plat |
  ($base * $plat)
  | .permissions.allow = (($base.permissions.allow // []) + ($plat.permissions.allow // []))
  | .permissions.deny  = (($base.permissions.deny  // []) + ($plat.permissions.deny  // []))
' settings.base.json settings.{mac,windows}.json > ~/.claude/settings.json
```

Unpacked:

- `.[0] as $base | .[1] as $plat` — name the two inputs
- `$base * $plat` — recursive merge of the two objects (later wins for scalar keys; nested objects merged)
- Then **overwrite** `permissions.allow` and `permissions.deny` with the concatenation of the two arrays

The explicit concat step exists because jq's `*` operator replaces arrays on conflict; we want concatenation for the permission lists so the platform-specific allows add to the base instead of replacing it.

Result: the final `~/.claude/settings.json` has:
- All env vars from base (and platform, if any)
- `effortLevel` from base
- `extraKnownMarketplaces` from base
- `permissions.allow` = base allows + platform allows
- `permissions.deny` = base denies + platform denies

Accumulated grants from daily work get wiped on every re-install. That's intentional — see [10-design-decisions.md](10-design-decisions.md) for the reasoning.

## The `install.sh` execution flow

```
1. Export MSYS=winsymlinks:nativestrict (no-op on Mac/Linux)
2. Resolve DOTFILES = absolute path of the repo
3. Set TS = current timestamp
4. mkdir -p ~/.claude/
5. For each of CLAUDE.md, plan-mode.md, brainstorm-mode.md, commands/, plans/, brief-templates/:
   - If the target already exists and is not the symlink we want,
     move it to {target}.backup.{TS}
   - Create the symlink
6. Detect platform from uname -s
7. If jq present:
   - Backup existing ~/.claude/settings.json if any
   - Run the merge expression, write result
8. Generate copilot-prompts/claude-global.instructions.md from CLAUDE.md,
   copilot-prompts/plan-mode.instructions.md from plan-mode.md, and
   copilot-prompts/brainstorm-mode.instructions.md from brainstorm-mode.md
9. Detect VS Code user dir based on platform
   - If found, loop over all copilot-prompts/*.md and symlink each into it
10. Pick shell rc (prefer .bashrc when both exist, else .zshrc)
    - If not already present, append PATH update for bin/
11. chmod +x bin/* (best effort; fails silently if denied)
12. Run smoke tests (check each expected symlink, check settings)
13. Print "Next steps" message
```

`install.sh` is idempotent: running it twice is safe. Everything it does is a no-op if the target is already correct.

## The `preflight.sh` execution flow

`preflight.sh` is read-only. It runs these nine checks:

```
1. Platform detection (uname -s)
2. Required tools present (git, jq, ln, readlink, chmod, mkdir, grep)
3. Symlink capability (create a real symlink in /tmp, verify [ -L ] passes)
4. All 18 expected repo files present
5. What install.sh would back up in ~/.claude/
6. Existing settings.json analysis (if jq available)
7. VS Code user dir + existing prompts directory state
8. Shell rc file detection
9. Git user config
```

Each check is `ok()`, `bad()`, or `warn()`. Exit 0 if no failures (warnings are fine), exit 1 if any failures.

## What's synced between machines

Everything in the repo is synced via git. Specifically:

**Synced:**
- `CLAUDE.md`
- `plan-mode.md`
- `brainstorm-mode.md`
- All slash commands in `commands/`
- All brief templates in `brief-templates/`
- All Copilot prompts in `copilot-prompts/` (except the generated instructions file)
- All plans in `plans/` — write on one machine, `git pull` on another, visible via symlink
- All settings files (`settings.base.json`, `settings.mac.json`, `settings.windows.json`)
- `install.sh`, `preflight.sh`, `bin/claude-handoff`
- All docs

**Not synced (machine-local):**
- `~/.claude/settings.json` — regenerated per-install, contains platform-specific merges and may accumulate one-shot grants between installs
- `~/.claude/projects/` — per-project conversation history and memory
- `~/.claude/.credentials.json` (if Claude Code uses one)
- `~/.claude/cache/`, `~/.claude/logs/`, `~/.claude/statsig/` — runtime state
- VS Code `settings.json` itself — deliberately not touched
- `copilot-prompts/claude-global.instructions.md` — gitignored; regenerated from `CLAUDE.md` on each install
- `copilot-prompts/plan-mode.instructions.md` — gitignored; regenerated from `plan-mode.md` on each install
- `copilot-prompts/brainstorm-mode.instructions.md` — gitignored; regenerated from `brainstorm-mode.md` on each install

## Cross-machine consistency guarantee

After `git pull && bash install.sh` on both machines, the following are **identical byte-for-byte**:

- `CLAUDE.md` content
- `plan-mode.md` content
- `brainstorm-mode.md` content
- Every file in `commands/`
- Every file in `brief-templates/`
- Every file in `copilot-prompts/` (including the regenerated one, since the source is identical)
- Every file in `plans/`

The following are **semantically identical but may differ slightly in bytes**:

- `~/.claude/settings.json` — same base content, but platform additions differ; mac vs windows vs linux will each have slightly different allow lists
- The local `additionalDirectories` or accumulated allows (if any) — machine-local

The following are **intentionally different per machine**:

- Absolute paths in the install output
- Timestamps on backup files
- VS Code user directory paths
- Per-project memory (`~/.claude/projects/`)

## Intercom subsystem

Intercom uses an MQTT broker to route prompts from a dispatcher machine to a receiver machine, with replies flowing back and surfacing in the dispatcher's Claude Code session.

```
DISPATCHER (Claude Code session)
  |
  |  ~/bin/send-job → mosquitto_pub
  |  topic: jobs/<machine>/<repo>
  ▼
BROKER (mosquitto)
  |
  |  mosquitto_sub
  ▼
RECEIVER (Mac mini or Linux)
  |  intercom-receiver (from claude-intercom/receiver/)
  |  runs: claude -p "$prompt"
  |  mosquitto_pub topic: replies/<machine>/<repo>
  |
  ↓ (back through broker)
  |
DISPATCHER inbox
  |  ~/bin/intercom-inbox-listener (Task Scheduler, Windows)
  |  appends to ~/.local/state/intercom/inbox.jsonl
  |
  |  UserPromptSubmit hook
  |  ~/.claude/hooks/surface-intercom-replies.sh
  ▼
  Claude Code response (unread replies prepended)
```

**claude-config's role** — dispatcher-side setup only:

```
claude-config/
  bin/                              ← dispatcher helpers; symlinked → ~/bin/
    send-job
    intercom-session
    intercom-machines
    intercom-repos
    intercom-inbox-mutate
    intercom-inbox-listener
  hooks/
    surface-intercom-replies.sh     ← UserPromptSubmit hook; symlinked → ~/.claude/hooks/
  windows/
    intercom-inbox-listener.xml.template  ← Task Scheduler task (rendered at install)
  commands/
    register.md  send.md  draft.md  machines.md  repos.md
```

**claude-intercom's role** — receiver runtime, broker config, tests, docs. claude-config does not clone or pin this repo — helpers call `mosquitto_pub`/`mosquitto_sub` directly and are maintained by copying updated versions from claude-intercom into `bin/` here.

**Upgrade flow:** pull updated helpers from claude-intercom/dispatcher/windows/, copy into `bin/`, commit, push. On each machine: `git pull` (symlinks propagate immediately, no re-install needed for in-place edits). Re-run `install.sh` only when new helpers are added or install.sh itself changes.

See [intercom-runbook.md](intercom-runbook.md) for the ops reference and [the product repo's dogfooding guide](https://github.com/danrichardson/claude-intercom/blob/main/docs/dogfooding-guide.md) for day-to-day usage.

## Common gotchas

**Editing `~/.claude/commands/ticket-new.md` vs editing `~/src/claude-config/commands/ticket-new.md`.** They're the same file because of the symlink. Either path works. Git sees the edit because the underlying file is in the repo.

**Editing `~/.claude/settings.json`.** Don't. Your edit will be wiped the next time `install.sh` runs. If you want to add a permission, add it to `settings.base.json` (universal) or `settings.{mac,windows}.json` (platform-specific), commit, push, re-install.

**Editing `copilot-prompts/claude-global.instructions.md`, `copilot-prompts/plan-mode.instructions.md`, or `copilot-prompts/brainstorm-mode.instructions.md` directly.** Don't. They're regenerated from their source files on every install. Edit the sources (`CLAUDE.md`, `plan-mode.md`, `brainstorm-mode.md`) instead.

**Deleting `~/.claude/plans/_next.md`.** Fine — it's just a pointer file created by `claude-handoff`. It gets recreated the next time you run handoff.

**Deleting `~/.claude/commands.backup.*` after install.** Fine if the original is safely in the repo (check `ls ~/src/claude-config/commands/` to confirm). Keep the `settings.json.backup.*` longer — it's the only record of any accumulated grants you might want to mine.
