# Install

Detailed walkthrough for setting up `claude-config` on a new machine (or re-installing on an existing one). For the quick checklist version, see [06-adding-a-new-machine.md](06-adding-a-new-machine.md).

## Prerequisites

You need:

- **git** — for cloning and pulling the repo
- **bash** — for running `install.sh` and `preflight.sh` (Git Bash on Windows works)
- **jq** — for merging the settings JSON files
- **Claude Code** installed on the machine where you want Claude Code to work
- **VS Code** (optional, but needed if you want Copilot Chat integration)
- **GitHub Copilot** extension in VS Code (optional)
- **Ability to create symlinks** — on Mac/Linux this is built in; on Windows it requires **Developer Mode** (Settings → System → For developers → Developer Mode: **On**) or running Git Bash as administrator

### Installing jq

| Platform | Command |
|---|---|
| macOS | `brew install jq` |
| Ubuntu/Debian | `sudo apt install jq` |
| Fedora | `sudo dnf install jq` |
| Windows (winget) | `winget install jqlang.jq` |
| Windows (scoop) | `scoop install jq` |

After installing jq on Windows, **close and reopen Git Bash** so that `which jq` finds the new binary.

## Mac / Linux

The happy path:

```bash
git clone git@github.com:<you>/claude-config.git ~/src/claude-config
cd ~/src/claude-config
bash preflight.sh
bash install.sh
source ~/.zshrc      # or ~/.bashrc
```

Verify:

```bash
ls -la ~/.claude/CLAUDE.md ~/.claude/plan-mode.md ~/.claude/brainstorm-mode.md ~/.claude/commands ~/.claude/plans ~/.claude/brief-templates
# Should show six symlinks pointing into ~/src/claude-config/

jq .effortLevel ~/.claude/settings.json
# Should print "max"

which claude-handoff
# Should print /Users/you/src/claude-config/bin/claude-handoff
```

## Windows (Git Bash)

Windows has three gotchas that Mac/Linux don't. `preflight.sh` catches all of them, so **always run preflight first** on Windows.

### Step 1: enable Developer Mode

Settings → System → For developers → Developer Mode → **On**. This allows non-admin users to create symbolic links. No reboot required. Leave it on permanently; it has no other practical effect.

### Step 2: install jq

```powershell
winget install jqlang.jq
```

Then close and reopen Git Bash so the new binary is on `PATH`.

### Step 3: clone and preflight

In Git Bash:

```bash
cd ~
mkdir -p src && cd src
git clone git@github.com:<you>/claude-config.git
cd claude-config
bash preflight.sh
```

Read the preflight output carefully. Expect to see:

- `[1] Platform`: Windows shell detected (MINGW64)
- `[2] Required tools`: all present (if jq is missing, you forgot step 2)
- `[3] Symlink capability`: ✓ (if ✗, see the troubleshooting section below)
- `[4] Repo contents`: all 22 expected files present
- `[5] What install.sh would back up`: warnings about any existing `~/.claude/` content
- `[6] Existing settings.json analysis`: report of any accumulated allow grants (will be regenerated by install.sh)
- `[7] VS Code (Copilot) detection`: VS Code user dir present
- `[8] Shell rc`: `~/.bashrc` found
- `[9] Git configuration`: git user name + email set

### Step 4: install

```bash
bash install.sh
```

`install.sh` automatically exports `MSYS=winsymlinks:nativestrict` before creating symlinks, so you don't need to set it yourself. If the script reports any smoke-test failures, **stop and read the error** — don't re-run install.sh blindly.

### Step 5: pick up the new PATH

```bash
source ~/.bashrc
which claude-handoff
```

Should print the path to the `claude-handoff` script in the repo's `bin/`.

### Step 6: VS Code Copilot verification

Open VS Code. Open Copilot Chat (sidebar icon or `Ctrl+Alt+I`). Click "New Chat" to start a fresh conversation. Say:

```
send me a test prowl
```

If a Prowl notification arrives on your phone, the entire chain works: the instructions file was loaded, Copilot read the Prowl section from your `CLAUDE.md`, the model emitted the curl, the curl reached Prowl, Prowl pushed the notification.

### If symlinks still don't work after Developer Mode

If preflight's `[3]` is still `✗` after you've enabled Developer Mode:

1. **Close and reopen Git Bash.** Some Git Bash installs cache the symlink capability check at startup.
2. **Manually set MSYS for the shell:**
   ```bash
   export MSYS=winsymlinks:nativestrict
   bash preflight.sh
   ```
   This should turn `[3]` green. If it does, persist the export:
   ```bash
   echo 'export MSYS=winsymlinks:nativestrict' >> ~/.bashrc
   ```
3. **Check whether you're accidentally in WSL.** `uname -s` should print `MINGW64_NT-…` on Git Bash, or `Linux` on WSL. If you're in WSL by accident, either exit WSL and use Git Bash, or know that WSL installs into the WSL filesystem (different `~/.claude/`) not the Windows filesystem.
4. **Try running Git Bash as Administrator** for the one `bash install.sh` invocation. The symlinks created persist after you close the elevated shell.
5. **Last resort**: see [08-troubleshooting.md](08-troubleshooting.md) for the fallback of manually creating symlinks via PowerShell.

## What `install.sh` actually does

Run it with `-x` (`bash -x install.sh`) if you want to see every step, or read the script — it's ~150 lines. The summary:

1. **Export `MSYS=winsymlinks:nativestrict`** on Windows (no-op on Mac/Linux).
2. **Create `~/.claude/`** if it doesn't exist.
3. **Backup and symlink** six paths from the repo into `~/.claude/`:
   - `CLAUDE.md`
   - `plan-mode.md`
   - `brainstorm-mode.md`
   - `commands/`
   - `plans/`
   - `brief-templates/`
   If any of these already exists at the target, it's moved to `*.backup.{timestamp}` before the symlink is created. Nothing is deleted.
4. **Detect platform** (`Darwin` → mac, `Linux` → mac for settings purposes, `MINGW*`/`MSYS*`/`CYGWIN*` → windows).
5. **Merge settings.json** using jq: combine `settings.base.json` with `settings.{platform}.json`, concatenate the `permissions.allow` and `permissions.deny` arrays, write the result to `~/.claude/settings.json`. Backs up the existing file first.
6. **Generate** Copilot instruction files by prepending `---\napplyTo: "**"\n---\n\n` frontmatter to each mode doc: `copilot-prompts/claude-global.instructions.md` from `CLAUDE.md`, `copilot-prompts/plan-mode.instructions.md` from `plan-mode.md`, and `copilot-prompts/brainstorm-mode.instructions.md` from `brainstorm-mode.md`. All three generated files are gitignored — the source files are the single source of truth.
7. **Symlink every file in `copilot-prompts/`** into VS Code's user prompts directory if VS Code is detected (loops over all `*.md` files, so new prompts are picked up automatically on re-install):
   - Mac: `~/Library/Application Support/Code/User/prompts/`
   - Linux: `~/.config/Code/User/prompts/`
   - Windows: `%APPDATA%/Code/User/prompts/`
8. **Add `bin/` to PATH** by appending a line to `~/.zshrc` or `~/.bashrc` (whichever exists; prefers `.bashrc` when both are present).
9. **Run smoke tests**: verify all 6 symlinks exist, verify two specific files are visible via the symlinks, verify `jq .effortLevel` returns `"max"`.
10. **Print next steps**: remind you to `source` your rc file, show the `gh repo create` command, describe the Copilot smoke test.

## What `install.sh` does NOT do

- **Does not push to any remote.** You create the GitHub repo and push manually.
- **Does not install jq**, git, VS Code, or Claude Code. Those are prerequisites.
- **Does not touch `~/.claude/projects/`** (per-project memory, conversation history). That stays machine-local.
- **Does not touch VS Code `settings.json`.** The Copilot wiring uses the instructions file mechanism; no JSON edits are needed.
- **Does not modify any existing project's files.** Per-project bootstrapping happens via `/ticket-install` inside each project, not via this global installer.
- **Does not auto-update the repo.** You run `git pull` yourself (or fold it into `sync-repos`).

## Intercom bootstrap (MQTT era)

`install.sh` always runs an intercom-setup block after the Copilot wiring:

1. Symlinks every file in `bin/` into `~/bin/` using the same idempotent `link()` helper. This puts `send-job`, `intercom-session`, `intercom-machines`, `intercom-repos`, `intercom-inbox-mutate`, and `intercom-inbox-listener` on a well-known absolute path that the slash commands rely on.
2. Installs `hooks/surface-intercom-replies.sh` to `~/.claude/hooks/surface-intercom-replies.sh` (symlink). This hook is registered in `settings.base.json` as a `UserPromptSubmit` hook.
3. Checks for `mosquitto_pub` on PATH. If missing, prints install hints and continues — the symlinks are still in place for when mosquitto is installed later.
4. Checks for `~/.config/intercom/creds`. If absent and stdin is interactive, prompts for `MQTT_HOST`, `MQTT_PORT` (default 1883), `MQTT_USER`, `MQTT_PASS` and writes the file (chmod 600). Skipped in non-interactive mode.
5. **Windows only**: renders `windows/intercom-inbox-listener.xml.template` → `windows/intercom-inbox-listener.xml.rendered` (substituting `$USERNAME`) and registers the Task Scheduler task via `schtasks /Create /XML ... /TN intercom-inbox-listener /F`.

Prerequisites:

- **mosquitto-clients** — `brew install mosquitto` (Mac) / `winget install cedalo.mosquitto` (Windows) / `sudo apt install mosquitto-clients` (Linux). Without this, helpers are installed but won't connect to the broker.
- **MQTT broker** — accessible from the machine (Tailscale IP or direct hostname). You provide the coordinates at the creds prompt.

For creds rotation, Task Scheduler management, hook troubleshooting, and teardown, see [intercom-runbook.md](intercom-runbook.md).

## Re-installing (safe at any time)

`install.sh` is idempotent. Re-running it:

- Leaves existing symlinks alone if they already point to the right place
- Replaces symlinks if they point somewhere else (backing up the old target)
- Regenerates `~/.claude/settings.json` from the current `settings.base.json + settings.{platform}.json`
- Regenerates `copilot-prompts/claude-global.instructions.md` from the current `CLAUDE.md`
- Regenerates `copilot-prompts/plan-mode.instructions.md` from the current `plan-mode.md`
- Regenerates `copilot-prompts/brainstorm-mode.instructions.md` from the current `brainstorm-mode.md`
- Re-runs the smoke tests

You should re-run it whenever you:

- Edit `CLAUDE.md`, `plan-mode.md`, or `brainstorm-mode.md` (to regenerate the Copilot instructions files)
- Edit `settings.base.json` or a platform settings file (to regenerate `~/.claude/settings.json`)
- Pull changes that touch either of the above

You do **not** need to re-run it when you:

- Edit a file in `commands/` or `brief-templates/` (those are symlinked, so edits are live immediately)
- Edit a plan in `plans/` (same reason)

See [07-editing-and-syncing.md](07-editing-and-syncing.md) for the full "when do I need to re-install" matrix.
