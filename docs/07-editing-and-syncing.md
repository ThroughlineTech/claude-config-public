# Editing and Syncing

How to change things in this repo and propagate the change across machines. Short version: edit in Layer 1, commit, push, pull on the other machine, sometimes re-run install.sh.

## The edit-and-sync matrix

| I edited… | On the machine where I edited, I need to: | On other machines, after `git pull`, I need to: |
|---|---|---|
| `CLAUDE.md` | Re-run `install.sh` (regenerates Copilot instructions file) | Re-run `install.sh` |
| `plan-mode.md` | Re-run `install.sh` (regenerates Copilot plan-mode instructions file) | Re-run `install.sh` |
| `brainstorm-mode.md` | Re-run `install.sh` (regenerates Copilot brainstorm-mode instructions file) | Re-run `install.sh` |
| `commands/*.md` | Nothing (symlinked — edits are live) | Nothing |
| `brief-templates/*.md` | Nothing | Nothing |
| `copilot-prompts/run-brief.prompt.md` | Nothing | Nothing |
| `settings.base.json` | Re-run `install.sh` (regenerates ~/.claude/settings.json) | Re-run `install.sh` |
| `settings.mac.json` | Re-run `install.sh` on Mac | Re-run `install.sh` on Mac machines |
| `settings.windows.json` | Re-run `install.sh` on Windows | Re-run `install.sh` on Windows machines |
| `install.sh` | Re-run it (to apply any new behavior) | Re-run `install.sh` |
| `preflight.sh` | Nothing (just a script) | Nothing — pick it up on next run |
| `bin/claude-handoff` | Nothing (symlinked via PATH through bin/) | Nothing |
| A file in `plans/` | Nothing (symlinked) | Nothing |
| A file in `docs/` | Nothing (docs only) | Nothing |

**Rule of thumb**: if the change is to a file that's symlinked into Layer 2, it's live. If the change is to a file that's regenerated by `install.sh` (`~/.claude/settings.json`, `copilot-prompts/claude-global.instructions.md`, `copilot-prompts/plan-mode.instructions.md`, or `copilot-prompts/brainstorm-mode.instructions.md`), re-run `install.sh`.

## The canonical edit workflow

```bash
# 1. Edit the file in Layer 1
cd ~/src/claude-config
vi CLAUDE.md      # or commands/ticket-new.md, or whatever

# 2. Verify locally (if applicable)
# For command/brief/template edits: just use them, they're live
# For CLAUDE.md or settings edits: re-run install.sh first
bash install.sh

# 3. Commit and push
git add <files>
git commit -m "descriptive message about what and why"
git push

# 4. On the other machine(s)
cd ~/src/claude-config
git pull
bash install.sh   # only if the edit was to CLAUDE.md or a settings file
```

## Specific edit scenarios

### "I want to add a new rule to the global instructions"

Edit `CLAUDE.md`. Commit. Push. On other machines, pull and re-run install.sh (because the Copilot instructions file is regenerated from `CLAUDE.md`).

### "I want to add a new slash command"

Create a new file at `commands/ticket-whatever.md`. The file's content IS the command definition — Claude Code reads it verbatim. Commit, push, pull on other machines. No install.sh re-run needed because the `commands/` directory is symlinked.

See `commands/ticket-new.md` or `commands/ticket-investigate.md` for the format. The pattern is: `# Command Title`, then sections for Input, Pre-flight Checks, Steps, Rules.

### "I want to tweak how an existing command behaves"

Edit the command file in `commands/`. Commit, push. The edit is live on the edited machine because of the symlink; other machines pick it up after `git pull`.

### "I want a new brief template for a phase that doesn't exist yet"

Create `brief-templates/{phase}.md` with the same structure as existing templates (header for the human, role, ticket context, what to do, hard rules, when done). Update `commands/ticket-delegate.md` to know about the new phase (add it to the phase-specific preconditions table). Commit, push, pull.

### "I want to add a permission allow that I need on all machines"

Edit `settings.base.json`, add the entry to `permissions.allow`. Commit, push. Re-run `install.sh` on every machine.

### "I want to add a permission allow that I only need on Mac"

Edit `settings.mac.json`. Commit, push. Re-run `install.sh` on Mac. Windows machines don't need to re-run (the entry wouldn't be picked up by their merge anyway).

### "I want to add a permission deny (safety rule) to all machines"

Edit `settings.base.json`, add the entry to `permissions.deny`. Commit, push, re-run `install.sh` everywhere.

### "I want to stop syncing some particular file"

Add it to `.gitignore` in the repo, then `git rm --cached <file>`, commit, push. Future edits to that file won't be tracked, and other machines will leave it alone.

### "I wrote a plan on the laptop and want to execute it on the desktop"

```bash
# On the laptop (where the plan was written to ~/.claude/plans/some-name.md)
claude-handoff                  # picks the most recent plan
# or
claude-handoff some-name.md     # picks a specific one
# The script copies it to _next.md, commits both, pushes

# On the desktop
cd ~/src/claude-config
git pull
# Then in any Claude Code session: "execute the plan in ~/.claude/plans/_next.md"
```

Or fold `git -C ~/src/claude-config pull` into `sync-repos` on the desktop so you don't have to remember.

### "I want to add a new file to the `bin/` directory"

Add the file to `bin/{script-name}`, make it bash-compatible, commit, push. On each machine, re-run `install.sh` so it runs `chmod +x bin/*` (the symlinked directory is already on PATH from the first install).

### "I want to add or update docs"

Edit anything in `docs/` or the README. Commit, push. Docs are not functional; they propagate via git and are read directly out of the repo.

## What NOT to edit

- **Anything in `~/.claude/` directly.** Always edit in Layer 1 (`~/src/claude-config/`). Direct edits to Layer 2 either have no effect (if the file is symlinked, you're editing Layer 1 anyway but via a confusing path) or get wiped on next install (if the file is regenerated).
- **`copilot-prompts/claude-global.instructions.md`** — it's generated from `CLAUDE.md` on every install. Edit `CLAUDE.md` instead.
- **`~/.claude/settings.json`** directly — it's regenerated. Edit `settings.base.json` / `settings.mac.json` / `settings.windows.json`.
- **Files in Layer 2's VS Code user prompts directory** — those are symlinks into Layer 1. Edit the Layer 1 files.

## Edit-time safety habits

### Run `preflight.sh` before `install.sh` on other machines

When you pull a significant change to `CLAUDE.md` or `install.sh`, run preflight first on the other machine before install. It's read-only and will tell you if anything on that machine is in a state that will cause install to misbehave.

### Test on one machine before pulling to the other

For non-trivial changes, run and test locally before pushing. It's much easier to debug an issue on the machine that made the change than to debug it on a machine that just pulled the change.

### Use descriptive commit messages

`CHANGELOG.md` is optional to maintain manually, but git log should tell a coherent story. If your commit messages are all "wip" and "fix", future-you won't remember what changed or why.

### Re-run `install.sh` after pulling if you're unsure

`install.sh` is idempotent. Re-running it costs about 3 seconds and guarantees your local state matches the repo. If in doubt, run it. Worst case, it's a no-op.

### Don't commit secrets

`CLAUDE.md` currently contains the Prowl API key. That's intentional for a private repo. But if you ever add a new secret (e.g. a Figma API key, an OpenAI key, anything), think twice before committing it. See [09-faq.md](09-faq.md) for the "should this be in a gitignored secrets file?" discussion.

## Debugging a syncing issue

If a change you made doesn't seem to have propagated to the other machine:

1. **On the edited machine**: `git log --oneline` — is the change actually committed?
2. **On the edited machine**: `git status` — is there a local uncommitted file?
3. **On the edited machine**: `git push` — was it actually pushed?
4. **On the target machine**: `git pull` — did the pull succeed?
5. **On the target machine**: `ls -la ~/.claude/{CLAUDE.md,commands,plans,brief-templates}` — are the symlinks intact?
6. **On the target machine**: if you edited `CLAUDE.md` or a settings file, did you re-run `install.sh`?
7. **On the target machine**: `cat ~/.claude/settings.json | jq .permissions.allow | grep <thing-you-added>` — did the merge pick up your addition?

99% of "it didn't propagate" is either an un-pushed commit, a missing pull, or a missed re-install.
