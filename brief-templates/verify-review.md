## What to do

1. Read the Acceptance Criteria and Files Changed.
2. Read the checklist carefully.
3. Critique the checklist against these questions:

   ### Completeness
   - Does each Acceptance Criterion have at least one corresponding checklist item that would actually verify it?
   - Are there code paths in Files Changed that the checklist doesn't exercise?

   ### Specificity
   - Is each step concrete enough that a human can follow it without guessing? ("Click X, see Y") versus vague ("test the feature")?
   - Does each step describe what to OBSERVE, not what the code does internally?

   ### Edge cases
   - Are obvious edge cases (empty input, invalid input, network failure, etc.) covered for the changed code?

   ### Regression coverage
   - Does the checklist include regression checks for adjacent features that share code paths with the changes?

   ### Cost
   - Is the checklist a reasonable size? A 200-step checklist for a 1-line change is bad. A 3-step checklist for a major refactor is also bad.

## Hard rules

- DO NOT modify the original Verification Checklist section — write into the new Peer Review section only
- DO NOT modify any source code
