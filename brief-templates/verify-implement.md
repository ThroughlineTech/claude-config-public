## What to do

1. Read the Implementation Plan and Acceptance Criteria above carefully.
2. Read the diff carefully. For each file changed, understand what changed and why.
3. (Optional but recommended) Read the surrounding context of changed files — not just the diff — to make sure changes fit cleanly with the rest of the file.
4. Critique the implementation against these questions:

   ### Plan adherence
   - Did the implementer complete EVERY item in the Implementation Plan? Walk through each plan item and check the diff for it.
   - Did the implementer add anything NOT in the plan (scope creep)?

   ### Acceptance criteria
   - Does the diff actually satisfy each acceptance criterion? Walk through them one by one.

   ### Correctness
   - Are there logic bugs you can spot statically? (Off-by-one, wrong operator, missing null check, wrong variable name, etc.)
   - Are there edge cases the diff handles wrong or doesn't handle at all?
   - Are error conditions handled properly at boundaries?

   ### Project rules
   - Does the diff violate any rules from the project's CLAUDE.md (line limits, naming, structure, etc.)?
   - Does it follow existing patterns in the codebase, or does it introduce a foreign style?

   ### Test coverage
   - Are there new code paths without test coverage?
   - Do the new tests actually test the behavior the ticket cares about, or are they trivial assertions?

## Hard rules

- DO NOT modify any source code

- DO NOT change branches or commit anything
