## What to do

### Phase 1: Investigate

1. Read the description and acceptance criteria above carefully.
2. Read every file in "Key source locations" end-to-end. Do not skim.
3. Search the codebase for additional files related to the ticket's domain.
4. For bugs: trace the code path and identify the root cause.
   For features: identify where new code fits, what interfaces need extending, what's needed.
5. Identify regression risks: what existing tests cover this area, what user-facing flows touch it.

Write your findings into the ticket file at `{TICKET_PATH}`:

**`## Investigation`** — what you found, with specific file paths and line numbers. Root cause or architectural fit. Regression risk (low/medium/high + why).

**`## Proposed Solution`** — your approach, why this over alternatives, tradeoffs.

**`## Implementation Plan`** — numbered checklist. Each step names the file, function/component, and exactly what changes. Be specific enough that someone reading only this plan could verify the implementation matches it.

### Phase 2: Implement

1. Verify you're on branch `{BRANCH}`. If not, stop and report.
2. Work through your Implementation Plan item by item:
   - Read every file before modifying it
   - Follow existing patterns in the codebase
   - Follow project rules from the "Project rules" section above
3. Add tests following existing conventions:
   - Happy path and error cases
   - Edge cases relevant to the change
   - If a test you didn't write fails, that's a regression — fix it
4. Run the Test command. ALL tests must pass.
5. Run the Build command. Must complete cleanly.
6. Run the Lint command if configured.

### Phase 3: Commit and document

1. Stage files specifically (`git add path/to/file`, NOT `git add -A`)
2. Commit with format: `{ID}: {short description}`
   - Multiple logical units → multiple commits, each prefixed with `{ID}:`
3. Push the branch: `git push -u origin {BRANCH}`
4. Update the ticket file at `{TICKET_PATH}`:
   - **`## Files Changed`** — list every file created or modified with a one-line description
   - **`## Test Report`** — tests added, total passing, build status, any warnings
   - Set `status: review`
   - Update the `updated:` date

## Hard rules
